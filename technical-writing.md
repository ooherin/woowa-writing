# 프론트엔드에서 웹 접근성을 높이는 방법

## 예상 독자

- 웹 접근성에 흥미가 있거나 알고 싶은 분
- 웹 접근성을 지키는 방법을 알고 자신의 서비스에 적용하고 싶은 프론트엔드 개발자

## 서론

웹 접근성은 장애를 가진 사람들을 포함한 모든 사용자가 웹사이트나 애플리케이션을 쉽게 이용할 수 있도록 보장하는 것을 목표로 합니다. 이는 단순히 디자인의 개선이나 기술적 기능의 추가를 넘어서, 모두가 동등하게 서비스에 접근할 수 있는 환경을 조성하는 것을 뜻합니다.

웹 접근성을 높이는 방법을 알아보기 전에 우선, 당신이 보고 싶은 영상을 검색하여 찾아보는 과정을 생각해 봅시다. 우선 비디오 앱을 클릭하여 접속한 화면을 '눈으로' 확인할 것입니다. 검색 창을 누르면 하단에 뜨는 키보드를 '보고' 쉽게 검색어를 입력할 수 있죠. 원하는 영상을 찾으면 옆에 있는 '마우스를 사용하여' 클릭하여 영상을 보면 끝입니다.

이 과정을 만약 눈이 안 보이는 상태로 진행한다면 어떨 것 같나요? 대부분의 서비스를 위의 방식대로 이용하기는 매우 어려울 것입니다.
이렇게 시력이 낮거나 특정 색을 인지하지 못하는 사용자는 이러한 '시각'에 기반한 기본적인 탐색 행위에 어려움을 갖습니다. 손 움직임에 제약이 있는 사용자는 '마우스 사용'에 기반한 행위에 제약을 받기도 합니다. 평범한 웹 서비스에서 다음과 같은 흔한 요소들은 해당 사용자들에게 다음과 같은 제약을 줍니다.

- 텍스트, 버튼 등 시각적인 모든 요소 - 시력이 안 좋은 사람에게 제약
- 대비가 거의 없는 색으로 이루어진 요소 - 적록색맹 등 색을 잘 구별하지 못하는 사람에게 제약
- 섬세한 마우스나 키보드 작업을 해야 하는 요소 - 손 운동이 어려운 사람에게 제약

이들은 일반적인 상황과는 다른 제약을 가진 상황에 처해 있는 사용자들입니다. 우리는 이런 사용자들이 사용자의 일부분을 차지한다는 것을 쉽게 잊고는 합니다. 이들도 무리 없이 웹 페이지를 사용할 수 있어야 하며, 그러한 환경을 조성하는 것이 웹 접근성의 목표라고 할 수 있습니다.

서비스를 만들 때 웹 접근성을 고려하는 행위는 오직 장애가 있는 사람들만을 위한 것이 아닙니다. 놀랍게도, 장애가 없는 사람들도 더 편리해집니다. 글 초반에 언급한 웹 접근성의 목표에서 '모두가 동등하게 서비스에 접근할 수 있는 환경'에 '모두'가 들어가는 이유입니다.

예를 들어, 우리가 검색이나 입력할 때 편리하게 쓰는 자동완성 기능은 원래 장애를 가진 사용자들을 위한 기능이었습니다. 특히, 시각 장애나 손이 불편한 사람들에게 유용하도록 만들어졌습니다. 이런 사람들은 전반적으로 타이핑이 어렵거나, 시간이 오래 걸리는 문제가 있었죠. 하지만, 현재 자동완성 기능은 장애가 없는 사람도 빠르고 쉽게 타이핑하기 위해 자동완성 기능을 사용합니다. 이들도 특정 상황에서 이 기능의 편리함을 느끼기 때문입니다.
결론적으로, 장애가 있는 사용자를 고려하는 행위는, 다음 그림과 같이 곧 장애가 없는 사용자 또한 고려하는 행위라고 할 수 있습니다.

<img src="https://github.com/user-attachments/assets/c65a7933-0854-4973-8db7-bd28e3fb8c87"/>

우리나라의 <a href="http://www.kwacc.or.kr/Accessibility/Law" target="_blank">「장애인차별금지 및 권리구제 등에 관한 법률」</a>에서는 장애가 있는 사람도 비장애인과 동등하게 웹사이트를 이용할 수 있도록 웹 접근성을 준수할 것을 명시하고 있습니다. 법률에 따르면, 웹사이트와 애플리케이션의 설계 및 개발 시 접근성을 고려하지 않으면 차별 행위로 간주될 수 있으며, 이는 법적 책임으로 이어질 수 있습니다.

이 법률의 존재는 서비스에서 웹 접근성이 웹 서비스에서 얼마나 중요하게 여겨지는 가치인지 보여줍니다. 웹 서비스의 UI와 유저와의 상호작용을 담당하는 프론트엔드 개발자라면, 이러한 가치를 알고 어떻게 웹 접근성을 높일 수 있는지를 알아야 합니다.
이제 이 접근성을 어떻게 하면 높일 수 있을지 알아보도록 하겠습니다.

## 1. 웹 접근성을 높이는 기본적인 방법

웹 접근성을 올릴 수 있는 기본적인 방법들을 먼저 소개하겠습니다.

### (1) 언어 및 제목 설정

스크린 리더기는 낭독기이기 때문에, 언어의 영향을 받습니다. `index.html` 파일에 `lang` 을 `ko`로 설정하면 스크린 리더기의 언어를 한국어로 설정할 수 있습니다. 이는 접근성 뿐만 아니라 `SEO(검색 엔진 최적화)`에도 도움이 됩니다. 그리고 `title` 태그를 사용해 해당 문서에 대한 제목을 설정해 주세요. 스크린 리더기가 처음 웹을 탐색할 때 이 제목을 읽어주고, 사용자는 이 제목을 듣고 어떤 웹 페이지에 들어왔는지를 쉽게 알 수 있습니다.

```html
/*index.html*/
<!DOCTYPE html>
<html lang="ko">
  <head>
    <meta charset="UTF-8" />
    <title>웹 접근성을 올리는 방법</title>
  </head>
</html>
```

### (2) 시맨틱 태그 사용

접근성을 높이는 많은 방법들이 있지만, 시맨틱 태그를 활용하는 것은 가장 기본적이자 필수적인 방법입니다. 올바른 태그를 사용하면 내용의 가독성을 높여줄 뿐만 아니라, 사용자의 탐색이 쉬워지기도 합니다. 또한 `SEO(검색 엔진 최적화)` 에도 효과적입니다. 구글 같은 검색 엔진은 시맨틱 태그를 기반으로 페이지의 내용을 분석하고, 페이지 내의 키워드들의 우선순위를 파악하기 때문입니다. 즉, 올바른 시맨틱 태그를 사용하는 것은 접근성 외에도 많은 장점을 가지고 있기에 꼭 기억해 두시는 것이 좋습니다.

시맨틱 태그는 콘텐츠의 의미와 기능에 따라서 선택하는 것이 좋습니다. 그 콘텐츠가 어떤 역할을 하는지, 어떤 내용을 담고 있는지에 따라서 말이죠. 예를 들어, 페이지의 주요 콘텐츠나 본문은 `<main>` 으로 감싸는 것이 좋습니다. 한 페이지에 주제나 섹션이 나뉘어 분리가 필요하다면 `<section>` 을 사용하는 건 어떨까요?  
이 외에도 `<nav>`, `<article>` 등의 주요 시맨틱 태그들의 사용법을 알고 적용하는 것이 중요합니다. `<div>`를 무턱대고 쓰기 보다는 시맨틱 태그를 적절하게 사용하는 습관을 들여봅시다.

### (3) `<h1>` - `<h6>` 구획 제목 요소

`<h1>`-`<h6>`로 이루어진 HTML 구획 제목 요소를 적절하게 사용하는 것도 중요합니다. 이 태그들은 문서의 구문을 계층화시키는 시맨틱 태그의 일종입니다.

이 중에서 `<h1>` 태그의 중요성에 대해 얘기해보려고 합니다. `<h1>` 태그는 제목을 의미합니다. 그래서 제목 태그라고 불리기도 합니다. 책의 제목은 딱 하나만 존재하는 것 처럼, 웹 페이지의 `<h1>` 도 페이지 당 딱 하나만 존재해야 합니다. 또한, `<h1>`는 한 페이지 내에 하나를 넣는 것이 좋습니다. 구글 같은 SEO가 콘텐츠를 분석할 때 `<h1>`로 감싸진 제목을 가장 우선시하기 때문입니다. 이는 검색 기능에서 사용자가 적절한 콘텐츠를 찾을 때도 사용됩니다. 마치 저희가 책을 고를 때 제목을 보고 고르는 것처럼 `<h1>`은 콘텐츠 탐색에서 핵심적인 역할을 합니다.

구획 요소에서 쉽게 실수할 수 있는 것이, 구획 요소의 순서를 바로 건너 뛰는 것입니다. `<h1>` 태그의 다음 요소에 구획 제목 요소를 사용하고 싶다면 `<h3>` 이 아닌 `<h2>` 를 써야만 합니다. 만약 스타일 때문에 이렇게 태그를 뛰어 넘었다면 이는 잘못된 것입니다. `<h1>`-`<h6>` 은 스타일만을 위한 태그가 아닌, 그 이상의 구문적 의미를 가지고 있기 떄문입니다. 이렇게 접근성을 높히려다 오히려 망치는 안티 패턴을 알고 피하는 것도 중요합니다.

## 2. 스크린 리더 접근성을 높이는 방법

스크린 리더는 스크린의 정보를 음성으로 출력해 주는 화면 낭독 프로그램입니다. 스크린 리더는 화면에 출력된 텍스트, 이미지, 버튼 등을 음성으로 변환해줍니다. 저시력 시각장애인과 난독증을 가진 사람들은 쉽게 글씨를 읽을 수 없기 때문에 이 스크린 리더을 사용해 웹을 탐색합니다. 스크린 리더 접근성을 높이는 것은 스크린 리더를 사용하는 사용자에게 좋은 환경을 만드는 것입니다. 그렇다면 스크린 리더 접근성을 올리기 위한 방법으로는 어떤 것들이 있을까요?

### ARIA

ARIA 는 `Accessible Rich Internet Applications` 의 약자로, 장애를 가진 사용자가 웹 사이트를 쉽게 접근할 수 있도록 돕는 여러 속성들을 말합니다. ARIA는 HTML 에서 표현할 수 없는 접근성을 표현하여 HTML 을 보충해줍니다. ARIA는 역할(role), 속성(property), 상태(state) 로 이루어져 있습니다. 사실 3개의 분류를 중요하게 나누기 보다, ARIA를 어떤식으로 활용할 수 있는지가 더 중요합니다.

### (1) 역할 (role)

`role` 속성은 HTML 이 어떤 역할을 하는지 정의합니다. 이 속성은 스크린 리더가 해당 요소가 특정 역할 요소(버튼, 체크박스, 검색) 으로 인식하게 합니다. 보통 이 속성은 태그로 표현할 수 없는 상황에서 제한적으로 사용합니다.

예를 들어, 밑 예제에서는 form 태그를 사용하여 상품 검색 바를 구현했습니다. 하지만 스크린 리더 사용자는 이 form 태그가 어떤 역할을 하는 form 인지 알 수 없습니다. form은 검색 기능, input을 채워서 제출하는 기능 등 다양한 용도로 쓰이기 때문입니다. 이 경우, `role=”search”` 를 추가해 사용자에게 힌트를 줄 수 있습니다. 스크린 리더는 `role=”search”` 가 적용된 요소에 포커스가 가면, “검색" 이라고 사용자에게 알려줍니다. 이걸 듣는 사용자는 쉽게 그 요소의 기능을 파악할 수 있습니다.

```jsx
/* role 속성을 사용 */
<form role="search">
  <label htmlFor="search-input">Search:</label>
  <input type="text" id="search-input" name="search" placeholder="상품  검색" />
  <button type="submit">검색</button>
</form>
```

이 `search` 외에도 `dialog`(모달 대화 상자), `banner`(페이지의 주요 헤더), `alert`(사용자에게 중요한 정보를 알릴 때 사용. 스크린 리더는 즉시 읽어줌.), `button`, `checkbox` 등 다양한 역할이 있습니다. 스크린 리더 사용자에게 시맨틱 태그 이상의 정보를 제공한다는 점에서, 분명 유용한 속성이지만 남발하면 자칫 접근성을 해칠 수 있습니다. 이에 대한 자세한 설명은 'ARIA 를 사용할 때 주의사항' 파트에서 다시 설명드리겠습니다.

### (2) 속성 (Properties)

두번째 ARIA의 구조는 속성입니다. 상황에 따라 role 보다 스크린 리더에 많은 내용을 전달할 수 있는 기능입니다. 속성은 3번째 요소인 상태(state)와 혼동하기 쉬운데, 서비스 내에서 자주 변경되는 것은 상태(State), 그렇지 않은 것은 속성(Properties)이라고 이해하시면 좋습니다.

#### aria-label

`aria-label` 은 스크린 리더가 읽어주는 요소의 이름이라고 할 수 있습니다. 요소 안에 text 만으로 설명이 충분하지 않을 때 이 `aria-label` 을 사용하면 좋습니다. 예를 들어, 항공사 페이지를 만드는 도중 다음과 같은 컴포넌트가 있다고 생각해볼까요?

<img width="333" alt="스크린샷 2024-10-29 오후 2 19 11" src="https://github.com/user-attachments/assets/f56bd1b4-d55d-40ad-9b44-6ee00ae4ce86">

위의 파란색 박스의 컴포넌트는 다음과 같이 구현된 상태입니다.

```jsx
<label>성인</label>
/*증가 버튼*/
<button
  className="button-text"
  onClick={incrementCount}
  type="button"
>
  <img src={plus} alt="증가 아이콘" />
</button>
...
```

딱 보기에는 아무 문제가 없어 보일지도 모릅니다. 하지만 시각장애인들은 버튼이 '성인 인원을 바꾸는 버튼' 이라는 사실을 인지하기가 힘듭니다. 왜 그럴까요?

시각장애인들은 UI를 통해 알 수 있는 정보를 인식할 수 없기 때문입니다. 즉, 버튼 바로 옆에 성인이라는 라벨이 있더라도, UI 를 볼 수 없기 때문에 라벨과 버튼과의 연관성을 알기가 쉽지 않습니다. 이 경우에는, 이 버튼이 어떤 역할을 하는지에 대해 `aria-label` 을 사용해서 직접 알려주어야 합니다. 밑처럼 aria-label=”성인 승객 증가” 를 추가한다면, 사용자가 버튼을 포커스 했을 때 스크린 리더가 이 버튼이 ”성인 승객 증가” 버튼이라고 읽어줄 것 입니다.

```jsx
/*사용자는 이 라벨과 버튼이 연결되어 있다는 것을 알기가 쉽지 않습니다.*/
<label>성인</label>
/*증가 버튼*/
/*스크린 리더는 이 버튼에 포커스가 가면 aria-label의 내용을 읽어줍니다.*/
<button
  aria-label='성인 승객 증가'className="button-text"
  onClick={incrementCount}
  type="button"
>
  <img src={plus} alt="증가 아이콘" />
</button>
```

#### 스크린 리더 전용 메시지

만약 사용자가 밑의 버튼을 클릭하면, 자신이 선택한 성인 승객이 몇 명이 되었는지 알 수 있을까요? 아마 쉽지 않을 것입니다. 저희가 제공한 승객 숫자 기능은 오직 시각적인 UI 로만 알 수 있습니다. 이를 보지 못하는 사용자는, 현재 성인 승객이 몇 명 선택되었는지 인지하기가 힘듭니다.

```jsx
<div>현재 선택된 성인 승객 수</div>;
/* 시각 장애인에게는 이 숫자가 보이지 않습니다. */
<span>{adultCount}</span>;
```

이 경우 가장 쉬운 방법은 UI 를 볼 수 없는 사용자만을 위한 상태 메시지를 만들어 주는 것입니다. 상태 메시지를 밑에다가 `<div>`, `<span>` 태그로 만들면, 스크린 리더는 렌더링 된 태그의 내용을 읽어주기 때문에 접근성이 향상 될 것 입니다.

하지만 태그를 사용하면 메시지가 UI에 그대로 노출되어 기존의 UI 를 해칠 수도 있습니다.
이렇게 스크린 리더 메시지를 위한 요소를 만들지만, UI에 노출시키고 싶지 않을 때 쓰는 것이 `visually-hidden` 기능입니다.`visually-hidden`를 사용한 요소는 보이지는 않지만, 이 요소들과 달리 렌더링 트리에 포함시킴으로써 스크린 리더가 해당 메시지를 읽을 수 있습니다. `visually-hidden` html의 요소는 아니고 개발자들이 자주 사용하는 CSS의 이름입니다. 이 CSS 속성은 검색을 통해, 쉽게 확인할 수 있으므로 여기에는 따로 넣지 않겠습니다.

만약 승객이 3명이 되었을 때 최대 승객 수에 도달했다는 스크린 리더 전용 메시지를 만들어 주고 싶다면 다음과 같이 하면 됩니다. state 로 메시지를 정의하고, 해당 메시지를 태그로 감싼 후 `visually-hidden` 클래스를 준다면, 해당 요소는 보이지 않지만 스크린 리더는 메시지를 잘 읽어줄 것입니다.

```jsx
const [statusMessage, setStatusMessage] = useState("");

const incrementCount = () => {
  if (adultCount === MAX_PASSENGERS) {
    /*최대 승객수에 도달하면, 스크린 리더가 다음 메시지를 읽습니다.*/
    setStatusMessage("최대 승객 수에 도달했습니다");
    return;
  }
  setStatusMessage("");
};

return (
  /*밑의 메시지는 UI에 노출되지 않습니다.*/
  statusMessage && (
    <div className="visually-hidden" role="alert">
      {statusMessage}
    </div>
  )
);
```

### (3) 상태 (status)

`aria-live` 와 같은 상태(State) 속성은 스크린 리더 등가 UI의 상태 변화를 감지하고 사용자에게 올바르게 전달할 수 있도록 도와줍니다. 웹 페이지는 상태 변화가 많이 일어나는데, 시각 장애인은 UI 를 볼 수 없기 때문에 이 변화를 인지하기가 힘듭니다. 따라서 이를 알려주기 위해 `aria-live`를 주로 사용합니다.

#### aria-live

aria-live는 `off`, `polite`, `assertive` 3가지의 상태가 있습니다. 기본적으로는 `off` 상태입니다. 이는 요소의 콘텐츠가 바뀌더라도 스크린 리더는 이를 사용자에게 읽어주지 않는 상태입니다. `polite`, `assertive` 는 콘텐츠가 변경되었을 때 읽어주는 속성들입니다. 하지만 이 둘의 차이는 해당 메시지의 우선순위입니다. `polite` 는 상태가 변경되어도, 스크린 리더가 진행 중인 내용을 막지 않습니다. 내용이 다 끝난 다음에 변경된 상태를 알려줍니다. 반면, `assertive` 는 해당 내용이 중요하다고 판단하여, 스크린 리더의 진행 작업을 중단하고 변경된 상태를 우선적으로 알려줍니다.

보통 `polite` 속성을 많이 쓰며, 에러 메시지와 같은 긴급한 내용일 경우에만 `assertive` 를 제한적으로 사용합니다. `assertive` 의 경우 스크린 리더의 작업 우선순위를 변경하므로, 작업 중인 사용자에게 혼란을 줄 위험이 있습니다. 따라서 `assertive` 는 필요한 곳에서만 제한적으로 사용해야 합니다.
밑의 예제코드와 같이 form에 에러 메시지에 사용할 수 있습니다.

```jsx
<form>
  <label for="name">이름/label>
  <input type="text" id="name" name="name">
  <button type="submit" onclick={handleSubmit}>제출</button>
</form>

/*에러 메시지가 생기면, 스크린 리더기는 이를 우선적으로 읽어줍니다.*/
<div id="error-message" role="alert" aria-live="assertive">
  이 항목은 필수로 입력하셔야 합니다.
</div>
```

이 `aria-live` 를 쓸 때는, 해당 속성을 남발하면 스크린 리더가 내용을 중복해서 읽을 수 있고, 이는 오히려 사용자에게 불편함을 줄 수 있습니다. 따라서 `aria-live` 를 쓰기 전에, 이 내용이 정말 사용자에게 필요한 내용인지 생각하고 사용해야 합니다.

#### ARIA 를 사용할 때 주의 사항

그렇다면 ARIA은 무조건 쓰면 좋은 기능일까요? 예상하셨다시피, 정답은 아닙니다. ARIA의 기능을 잘 못 사용한다면, 스크린 리더가 내용을 중복해서 읽어주거나, 불필요한 내용을 읽어서 오히려 사용자에게 불편함을 줄 수 있습니다. ARIA는 필요할 때만 사용해야 하며, 상당수가 HTML 태그로 대체될 수 있음을 인지하고 있어야 합니다. 개발자들이 ARIA 를 사용할 때 쉽게 실수하는 예제를 보여드리면서 설명드리도록 하겠습니다.

#### (1) 시맨틱 태그로 대체할 수 있는 경우

```jsx
/* good 시맨틱 태그를 적절하게 사용한 경우 */
<button onClick={handleClick}>버튼1</button>;

/* soso 태그에다가 role 속성을 사용한 경우 */
<div role="button" onClick={handleClick}>
  버튼2
</div>;
```

버튼1은 `<button>` 태그를 써서 만든 버튼이고, 버튼2는 `<div>` 태그에 `button` 이라는 `role` 을 주어 만든 버튼입니다. 겉보기에는 둘다 `onClick` 이라는 이벤트를 가져 영락없는 버튼처럼 보이나, 조금의 차이가 있습니다. `<button>` 태그의 경우 space, return, enter 키를 사용해 버튼을 클릭하는 기능이 내장되어 있습니다. 따라서 첫번째 버튼은, 키보드로 버튼에 접근하거나 enter로 버튼을 클릭할 수 있습니다.

하지만 두번째 버튼은 역할만 button 으로 지정해 놨을 뿐, 버튼 태그의 모든 기능을 사용할 수 없습니다. button 같이 키보드로 버튼에 접근할 수 있게 하려면 `tabIndex` 속성을 따로 지정하거나, keydown 이벤트를 직접 지정하는 등의 추가 작업이 필요합니다.

또 한 가지 알아두셔야 할 점은, ARIA 태그를 지원하는 환경은 제한적이라는 것입니다. 따라서 role은 해당 요소에 어울리는 시맨틱 태그를 먼저 알아보고 사용한 후, 적용할 시맨틱 태그가 없거나 마땅치 않을 때만 부가적으로만 사용하는 것이 좋습니다.

#### (2) 속성이 중복되거나 상충되는 경우

```jsx
/* bad 같은 속성을 중복 사용한 경우 */
<button role="button">...</button>
<fieldset role="group">...</fieldset>
<ul role="list">...</ul>

/* bad 체크박스의 상태가 서로 상충됨 */
<input type="checkbox" checked aria-checked="false" />
```

위 예제 또한 ARIA 속성을 잘 못 활용하고 있는 예시입니다. 첫 번째 예시는, 같은 속성을 중복해서 사용하고 있는 경우입니다. 이미 `<button>` 이라는 시맨틱 태그를 사용했는데, `role`로 `button`을 지정하는 것은 의미가 없습니다. 이는 오히려 코드의 가독성을 떨어뜨리고, 스크린 리더를 혼란스럽게 만들 뿐입니다. 많은 시맨틱 태그들은 이미 자기에게 맞는 ARIA 속성을 내재하고 있는 경우가 많습니다. 따라서 ARIA 속성을 사용할 때에는 이미 그 속성이 태그에 존재하지 않는지 확인해야 합니다.

두번째 예제는, 태그의 속성과 ARIA 의 속성이 상충되는 경우입니다. 첫 렌더링 때 `checked` 는 true이지만, `aria-checked` 에서는 false 속성을 가지고 있죠. 이렇게 되면 같은 `checked` 속성이 서로 상충됩니다. 굳이 `aria-checked` 를 쓸 필요가 없이 `checked` 속성만 지정해도, 스크린 리더는 이 체크박스의 체크 상태를 잘 읽어줍니다.
이렇게 추가한 ARIA 속성이 기존의 태그나, 상태와 상충되지 않는지도 확인해보면 좋습니다.

## 3. 키보드 접근성을 높이는 방법

마지막으로 키보드 접근성을 높이는 방법에 대해 설명 드리겠습니다. 키보드 접근성은 모든 사용자가 키보드만으로 웹 페이지와 상호작용할 수 있게 만드는 것입니다. 시각 장애인, 운동 장애인, 또는 키보드를 선호하는 사용자 등 다양한 사용자들이 마우스 없이 키보드만으로 웹사이트의 콘텐츠를 탐색하고 상호작용하는 것을 필요로 합니다.

### (1) tabIndex

키보드에서 요소의 포커스를 옮길 때 어떤 키를 쓸까요? 바로 탭 입니다. `tabIndex`를 사용하면, 개발자의 의도대로 사용자의 탭에 따른 포커스를 조정할 수 있습니다. `tabIndex` 의 순서는 `-1` => `0` => `양수` 순으로 작은 숫자부터 큰 숫자대로 이동하게 됩니다.

`tabIndex` 가 -1 일 경우에는, 요소가 포커스 되지 않습니다. `tabIndex` 가 0이면 기본적인 HTML 의 순서에 따라 요소가 포커스됩니다. `tabIndex` 가 1 이상이 되면 사용자가 순서대로 순서를 지정할 수 있습니다.

```jsx
<button tabindex="1">첫 번째로 읽음</button>
<button tabindex="2">두 번째로 읽음</button>
<button tabindex="0">세 번째로 읽음(HTML 기본 순서)</button>
<button tabindex="-1">읽지 않음</button>
```

하지만, 이렇게 `tabIndex` 를 양수로 지정하는 것은, 스크린 리더의 <a href="https://developer.mozilla.org/ko/docs/Web/HTML/Global_attributes/tabindex"> 의도적인 순서를 변경하는 것이므로 부작용이 생길 수 있어 가급적 사용하지 않는 것이 좋습니다.</a> 대신 요소들을 의미적으로 순서에 맞게 잘 배치하는 것을 추천합니다.

### (2) 본문 바로가기 기능

본문 바로가기 기능은 키보드 접근성을 높이는 좋은 방법입니다. 키보드 사용자는 상위에 링크부터 하나하나씩 이동하기 때문에, 원하는 본문 기능 까지 포커스를 하는 것이 오래 걸리기 때문입니다. 따라서 웹 사이트에 들어왔을 때, 바로 본문 위치까지 옮겨주는 본문 바로가기 링크를 걸어두면 좋습니다.

본문 바로가기는 가장 상단에 다음과 같이 만들어주면 됩니다. 바로 이동시키고 싶은 본문은 a 태그에 링크로 달아놓은 #content 에서 content 를 id 로 지정하면 됩니다.

```jsx
<a href="#content" className={styles.skipTo}>
  본문 바로가기
</a>

/*본문 바로가기를 선택하면 이 콘텐츠로 이동합니다.*/
<main id="content">
  //메인 콘텐츠
</main>

//css
.skipTo {
  position: absolute;
  top: -99px;
  left: 0;
  width: 100%;
  padding: 10px 0;
  text-align: center;
  text-decoration: none;
  z-index: 999;
}

.skipTo:hover,
.skipTo:focus,
.skipTo:active {
  top: 0;
}
```

해당 기능을 만들어주면, 키보드 사용자는 사이트에 처음 접근했을 때 탭을 누르면 숨겨져 있던 본문 바로가기 버튼을 볼 수 있습니다. 이 버튼을 선택하면 본문으로 바로 이동할 수 있습니다.

### (3) 포커스 트랩으로 포커스를 모달 내부에서만 돌게 만들기

모달은 접근성을 해치키 쉬운 요소입니다. 예를 들어, 모달이 열렸을 때 포커스는 기본적으로 모달의 내용이 아닌, 뒤에 본문 내용을 가리키게 됩니다. 사용자는 모달의 내용을 조작하고 싶은데 포커스가 모달 뒤의 내용에 있다면 곤란하겠죠.
이럴 때 사용하는 `포커스 트랩(FocusTrap)`이라는 기능이 있습니다. 포커스 트랩은 모달 안에서만 포커스가 이동하도록 만드는 것을 말합니다. 포커스 트랩의 경우 만드는 것이 조금 까다로울 수 있지만, 적용한다면 키보드 접근성을 크게 올릴 수 있는 방법이므로 추천합니다.

## 결론

이렇게 프론트엔드에서 접근성을 높이는 법을 알아봤습니다. 프론트엔드에서 접근성을 고려하는 것은 기능 몇 개를 추가하는 것 이상의 의미와 가치를 지닙니다. 웹 접근성을 높이면, 다양한 상황과 환경에 처한 사용자들이 차별 없이 서비스를 경험할 수 있게 됩니다. 제약이 있는 사용자들 뿐만 아니라, 제약이 없었던 사용자들의 서비스 경험도 더불어 향상시킵니다.

서비스 내에 제약이 있는 사용자들이 가진 벽을 하나씩 허물면서, 더 많은 사용자와 좋은 서비스 경험을 나눌 수 있다는 사실은, 제가 프론트엔드 개발자로서 색다른 행복을 느끼게 해주었습니다. 많은 프론트엔드 개발자들이 '웹 접근성'을 고려한 서비스를 개발하여, 모든 사람들이 제약 없이 원하는 서비스를 마음껏 누리는 날이 왔으면 좋겠습니다. 긴 글 읽어주셔서 감사합니다.
